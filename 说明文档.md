### 企业设备报修 Agent 概要设计

#### 一、 划分系统模块 (System Modules)

系统整体分为四大核心模块，解耦前端展示与后端 AI 逻辑：

1. **展现层：独立 Web Component 模块**

   - **核心职责：** 提供轻量级、跨平台（Web/H5/套壳小程序）的 UI 渲染 。负责设备硬件调用（麦克风录音、摄像头拍照）以及与后端的实时通讯交互 。

2. **接入与会话管理层：API Gateway & Session 模块**

   - **核心职责：** 处理前端的并发请求，管理多轮对话的上下文（Context）状态，并处理异常重试与限流策略。

3. **核心编排层：Agent Logic 模块**

   - **核心职责：** 系统的“大脑”。负责意图识别（普通对话、报修信息补充、转人工请求等）。根据对话状态，动态调用多模态大模型进行信息抽取，或组装最终的结构化 JSON。

4. **知识与模型服务层：RAG & LLM 模块**

   - 

     **核心职责：** 封装对外部大模型（文本+视觉）的调用接口。同时维护向量数据库，对用户输入的报修描述进行实时向量化，并召回最相似的历史工单数据以推断“故障类型”和“优先级” 。

#### 二、 主要技术选型 (Technology Stack Selection)

为保障轻量化接入与 AI 调用的高效性，推荐以下技术栈：

- **前端 Web Component：**

  - **框架：** 原生 JavaScript / TypeScript 或轻量级库 `Lit`（专门用于构建极小的 Web Components，打包后体积小，无框架冲突风险）。

  - 

    **通信与存储：** `WebSocket` 或 `SSE (Server-Sent Events)` 用于流式输出对话；`sessionStorage` 用于防丢失缓存 。

- **后端 API 与会话服务：**

  - **框架：** `Python FastAPI`（最适合 AI/算法服务的现代异步 Web 框架，高并发性能好）或 `Node.js (NestJS)`。

- **AI 大模型与 RAG 组件：**

  - **大模型 (LLM/VLM)：** 优先选择具备视觉识别能力的多模态大模型 API（如 GPT-4o、Claude 3.5 Sonnet，或国内的 Qwen-VL、智谱 GLM-4V）。
  - **向量数据库 (Vector DB)：** `Milvus`、`Qdrant` 或更轻量级的 `ChromaDB`（适合 5000 条级别的数据量且维护成本极低）。
  - **嵌入模型 (Embedding Model)：** BAAI/bge-large-zh 等优秀的开源中文 Embedding 模型。

#### 三、 数据库总体设计 (Database Design)

虽然我们的核心产出是“将 JSON 交给业务方” ，但我们自身系统仍需两类数据库来支撑业务运行：

1. **关系型数据库 (Relational DB - 如 PostgreSQL / MySQL)：**

   - **`session_logs` (会话日志表)：** 记录每一次对话的会话 ID、时间戳、原始对话文本、最终生成的 JSON。用于后续大模型的微调和系统迭代。

   - 

     **`fallback_records` (降级兜底表)：** 记录触发低置信度（`confidence: low`）或多次反问失败的异常 Case ，供研发分析大模型缺陷。

2. **向量数据库 (Vector DB - 如 Qdrant / Milvus)：**

   - 

     **`historical_tickets` (历史工单向量集合)：** 将 5000 条历史工单的“报修描述”进行向量化（Vector）存储 。

   - 

     **元数据映射 (Metadata)：** 每一条向量数据附带原始的 `fault_type`（故障类型代码）和 `repair_priority`（优先级），以便检索命中后直接提取使用 。

#### 四、 定义内外接口 (Interface Definitions)

接口设计严格遵循“零业务对接”原则 ：

**1. 外部接口（Agent 前端 -> 业务方 H5/小程序环境）** 这部分完全通过标准的浏览器原生事件（CustomEvent）实现，业务方只需在前端监听即可 ：

- 

  `onRepairTicketGenerated`：工单收集完成并确认后触发。`detail` 载荷即为你定义的完整结构化工单 JSON 数据 。

- 

  `onRequestHumanService`：用户主动点击或情绪降级时触发 。`detail` 载荷包含用户当前的半成品输入记录，业务方监听到后调起自身的客服弹窗 。

**2. 外部接口（前端 Web Component -> Agent 后端）**

- **`POST /api/v1/chat/init`**：初始化会话，获取 Session ID，建立身份验证（仅限组件安全鉴权，非用户身份）。
- **`POST /api/v1/chat/message`** 或 **WebSocket连接**：双向对话流。接收用户的文字/语音文件/图片，返回大模型的流式文本（Streaming）或追问指令。
- **`POST /api/v1/upload/image`**：图片直传接口，将用户拍照的图片上传至 OSS，返回图片 URL 供大模型分析。

**3. 内部接口（Agent 后端 -> RAG / LLM 服务）**

- 

  **`POST /internal/llm/vision-extract`**：传入图片 URL 和用户描述，调用 VLM 提取问题细节 。

- 

  **`POST /internal/rag/search-fault`**：传入用户描述提取的关键词，到 Vector DB 中执行 ANN（近似最近邻）搜索，返回匹配的 `fault_type` 和 `repair_priority` 。

### 一、 针对 5000 条历史工单的 RAG 索引重构策略

既然不能完全参考原始口语化描述，我们就需要在数据入库前（Data Ingestion阶段）增加一道“LLM 数据清洗”工序，剥离无效信息，提取核心特征。

针对 RAG 向量数据库，我建议采用如下的字段存储结构：

#### 1. 向量化字段 (Vector/Embedding Field)

这是用来跟当前用户输入的故障描述进行“计算相似度”的核心字段。

- 

  **不建议存：** 原始的 `报修描述` （太口语化，噪音多）。

- 

  **建议存（LLM预处理后的标准化文本）：** 在入库前，我们写一个脚本，让大模型把那 5000 条的 `报修描述`  统一转化为“物理实体 + 故障现象”的标准格式。

  - 

    *例如：* 原始数据“1102单元茶水间的会议室需要换一下锁芯”  -> 清洗并向量化的文本：“门锁损坏，锁芯需要更换”。将这个标准化后的纯净文本转化为向量存入。

#### 2. 元数据字段 (Metadata Fields / Payload)

这是当检索匹配成功后，向量数据库直接“带出”给大模型作为参考依据的结构化字段。不需要做向量化，直接存原值：

- 

  **`fault_type` (故障类型)：** 比如 `{"displayName":"土木工类报修","code":"400"}` 。这是最核心的指导数据 。

- 

  **`repair_priority` (报修优先级)：** 比如 `MEDIUM` 。帮助 Agent 结合历史判断轻重缓急 。

- 

  **`repair_type` (报修类型)：** 比如 `公司报修` 。

- **`raw_description` (原始报修描述)：** 作为附属参考字段保留（供后台排查时比对）。



**工作流变成：** 用户口述问题 -> Agent 提取“标准问题” -> 用“标准问题”去 RAG 搜索“标准问题” -> 命中后拉出对应的 `fault_type` 和 `repair_priority`  填入最终的 JSON。

### 二、 Agent 后端独立图片存储设计

既然确认由我们的 Agent 后端提供独立的图片存储方案，并在最终的 JSON 中包含生成的图片 URL ，我们需要在系统架构中落地以下设计：

#### 1. 技术选型

- **存储引擎：** 在我们的后端服务器上搭建轻量级的对象存储服务（如 **MinIO**），或者接入公有云的 OSS（如阿里云 OSS / 腾讯云 COS）。MinIO 对于独立部署的中间件来说是性价比极高、兼容 S3 协议的首选。（需要对图片做周期管理）

#### 2. 上传链路与安全

- 

  **API 接口：** 提供 `POST /api/v1/upload/image` 接口 。

- **安全机制：** Web Component 前端在获取到用户的照片后，携带 Session ID 向该接口上传图片。后端校验 Session 有效后接收文件，生成一个带有唯一哈希值的文件名（防止遍历攻击），并将其存储到指定的 Bucket 中。

- 

  **URL 回填：** 存储成功后，返回一个可公开访问（或带有时效性签名）的 HTTPS 图片 URL，Agent 会将这个 URL 装填入结构化 JSON 的 `image_urls` 数组中返回给前端业务方 。
