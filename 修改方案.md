# 修改方案

## 总览

按功能模块拆分为 4 个任务，建议按顺序实施（Task A 是其他任务的基础）。

---

## Task A — 字段扩展 + 工单 JSON 重构

**涉及文件：** `backend/app/agent/state.py`、`backend/app/agent/prompts.py`、`backend/app/agent/ticket_builder.py`

### A1. TicketDraft 新增字段

```python
estate: str | None = None    # 楼盘（如"前海嘉里中心"）
unit: str | None = None      # 单元号（如"1单元"、"A座"）
```

`missing_required()` 将 `estate`、`building`、`floor`、`unit` 全部列为必填。

### A2. EXTRACTION_SYSTEM 新增字段说明

```
- estate：楼盘/项目名称，如"前海嘉里中心"、"嘉里建设广场"、"万象城"
- unit：单元/座标识，如"1单元"、"A座"、"东塔"
```

### A3. reply_system_prompt 追问顺序

按优先级依次追问：
1. 缺问题描述 → 询问故障现象
2. 缺楼盘 → "请问您在哪个楼盘/项目？（如前海嘉里中心）"
3. 缺楼栋 → "请问您在哪栋楼？"
4. 缺楼层 → "在几楼呢？"
5. 缺单元号 → "请问是哪个单元/座？"
6. 全齐 → 进入确认

### A4. 工单 JSON 重构

最终输出格式：

```json
{
  "ticket_id": "71849302651847293",
  "repair_no": 1726198,
  "order_status": "COMPLETED",
  "repair_type": "公司报修",
  "location": {
    "estate": "前海嘉里中心",
    "building": "T25栋",
    "floor": "3楼",
    "unit": "1单元"
  },
  "problem_description": "空调不制冷",
  "image_urls": ["http://minio:9000/repairs/xxx.jpg"],
  "reporter": { "name": null, "phone": null },
  "visit_time": "5月22日 14时",
  "repair_priority": "HIGH",
  "fault_type": {
    "code": "001",
    "displayName": "机电类报修"
  }
}
```

字段说明：
- `ticket_id`：17 位随机数字字符串，`str(random.randint(10**16, 10**17 - 1))`
- `repair_no`：模块级计数器，初始值 1726198，每次 +1（重启后重置）
- `problem_description`：优先用 RAG 标准化描述，无结果时用用户原始描述
- `visit_time`：工单生成时刻，格式"M月D日 H时"
- 移除旧字段：`session_id`、`created_at`、`normalized_description`、`confidence`、`rag_match_score`

## Task B — 图片流程优化

**涉及文件：** `backend/app/agent/core.py`、`frontend/src/components/input-bar.ts`

### B1. 后端 — 图片追问

当所有必填字段已齐全但 `draft.image_urls` 为空时，在进入 RAG 检索前先追问：

> "请问您能提供一张现场照片吗？（可跳过，直接回复"跳过"）"

新增 `WAITING_IMAGE` 中间状态，等待下一条消息：
- 用户上传图片 → 收图后继续 RAG + 确认流程
- 用户回复"跳过"/"不用"/"没有" → 直接进入 RAG + 确认流程

### B2. 前端 — 相册/拍照二选一

当前 `<input type="file" capture="environment">` 强制调用相机。  
改为去掉 `capture` 属性，系统会自动弹出"拍照 / 从相册选择"选项（移动端标准行为）。

### B3. 前端 — 图片预览后再发送

用户选图后，在输入框上方显示缩略图预览（带 × 删除按钮），用户确认后点发送才上传并发送。

```
┌─────────────────────────────┐
│ [图片缩略图 ×]               │  ← 仅有待发图片时显示
├─────────────────────────────┤
│ 🎤  📷  [输入框............] ➤ │
└─────────────────────────────┘
```

新增 `_pendingImage: File | null` 状态。点发送时若有 `_pendingImage`，先上传图片再发送消息；文字和图片可同时发送。

## Task C — 语音输入改为填入文本框

**涉及文件：** `frontend/src/components/input-bar.ts`

### 当前行为

语音识别 `onFinal` 回调直接 dispatch `send-text` 事件，跳过文本框直接发送。

### 改动

将 `onFinal` 改为只填入 `_text`，不发送：

```typescript
onFinal: (text) => {
  this._interim = ''
  this._recording = false
  if (text.trim()) {
    this._text = text.trim()  // 填入输入框，用户可编辑后手动发送
  }
},
```

`onInterim` 同步更新 `_text` 显示实时识别内容，录音结束后用户可修改再发送。

## Task D — 模拟业务方页面重构

**涉及文件：** `frontend/index.html`

### 当前行为

收到 `onRepairTicketGenerated` 事件后，只在 `#event-log` 里打印一行 JSON 摘要。

### 改动

页面改为左右两栏布局：

```
┌──────────────────────────────────────────────────────────┐
│  业务方模拟页面                                            │
├──────────────────────┬───────────────────────────────────┤
│                      │  工单详情                          │
│  [聊天悬浮窗所在区域] │  位置：前海嘉里中心 T25栋 3楼 1单元 │
│                      │        [可点击编辑]                │
│                      │  报修描述：空调不制冷  [可编辑]     │
│                      │  图片：[img] [img]（直接展示）      │
│                      │  故障类型：机电类报修  [可编辑]     │
│                      │  上门时间：5月22日 14时  [可编辑]  │
│                      │                                   │
│                      │  [提交工单]                        │
└──────────────────────┴───────────────────────────────────┘
```

展示字段及来源：

| 展示项 | 来源字段 |
|--------|---------|
| 位置 | `location.estate / building / floor / unit` 拼接 |
| 报修描述 | `problem_description` |
| 图片 | `image_urls[]` → `<img>` 直接展示 |
| 故障类型 | `fault_type.displayName` |
| 上门时间 | `visit_time` |

所有展示项均可点击内联编辑（`contenteditable` 或 inline input）。  
"提交工单"按钮点击后显示"已提交"状态（纯模拟）。
