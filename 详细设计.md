```
└─────────────────────────────────────────────────────┘
```

### 1.2 核心数据流

```
用户开口说话
  → Web Speech API → 转文字 → POST /api/v1/chat/message
  → Session Manager 查找会话状态
  → Agent Core 调用 LLM 提取字段 (extraction pass)
  → 更新 TicketDraft（已收集字段集合）
  → 判断必填字段是否齐全
      ├─ 未齐全 → LLM 生成追问 → SSE 流式返回文字
      └─ 已齐全 → RAG 检索 fault_type / priority
                → LLM 生成确认摘要 → 等待用户确认
                → 用户确认 → 组装最终工单 JSON
                → SSE 推送 ticket_ready 事件
                → 前端触发 CustomEvent → 业务方
```

### 2.3 语音输入（Web Speech API）

```
用户长按 [说话] 按钮
  → SpeechRecognition.start()，lang="zh-CN"，interimResults=true
  → 实时显示灰色 interim 文字（给用户反馈）
  → 松开按钮 → SpeechRecognition.stop()
  → onresult 回调中取 final transcript
  → 填入输入框并自动发送

  → 检测到 !('webkitSpeechRecognition' in window)
  → 语音按钮变灰 + tooltip "请使用文字输入"
```

**TTS 输出（AI 语音播报）：**  
AI 回复文字通过 `window.speechSynthesis.speak()` 朗读，使用系统中文女声，语速 0.95。用户可在面板标题栏切换静音。

### 2.4 图片上传流程

```
用户点击 [拍照] 按钮
  → <input type="file" accept="image/*" capture="environment">
  → 获取 File 对象后，用 Canvas 压缩：
      最长边 ≤ 1920px，JPEG quality=0.85
  → 显示本地缩略图（objectURL）
  → POST /api/v1/upload/image (multipart + session_id)
  → 拿到 image_url 后，作为一条消息发给 Agent:
      { type: "image_url", content: "图片已上传", image_url: url }
  → Agent 将 URL 追加到 TicketDraft.image_urls
```

### 2.5 SSE 流式通信

前端用 `fetch` + `ReadableStream` 接收 SSE（避免 EventSource 不支持 POST body 的限制）：

```typescript
async function sendMessage(sessionId: string, msg: ChatMessage) {
  const resp = await fetch(`${apiBase}/api/v1/chat/message`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ session_id: sessionId, message: msg }),
  })
  const reader = resp.body!.getReader()
  const decoder = new TextDecoder()
  let buf = ''
  while (true) {
    const { done, value } = await reader.read()
    if (done) break
    buf += decoder.decode(value, { stream: true })
    for (const line of buf.split('\n\n')) {
      if (!line.startsWith('data:')) continue
      const payload = JSON.parse(line.slice(5).trim())
      handleSSEEvent(payload)   // 见下方事件处理
    }
    buf = ''
  }
}
```

**SSE 事件类型：**

| type | 含义 | payload |
|------|------|---------|
| `text_delta` | 流式文字片段 | `{ content: "好的" }` |
| `state_update` | 状态变化通知 | `{ state: "COLLECTING_LOCATION", collected: {...} }` |
| `ticket_ready` | 工单已就绪，待确认 | `{ ticket: { ...完整工单 JSON... } }` |
| `done` | 本轮响应结束 | - |
| `error` | 服务端错误 | `{ code: "LLM_TIMEOUT", message: "..." }` |

### 2.6 CustomEvent 向业务方回调

```typescript
// 工单确认完成
this.dispatchEvent(new CustomEvent('onRepairTicketGenerated', {
  bubbles: true,
  composed: true,   // 穿透 Shadow DOM
  detail: ticketJSON,
}))

// 用户点击"转人工"或触发情绪降级
this.dispatchEvent(new CustomEvent('onRequestHumanService', {
  bubbles: true,
  composed: true,
  detail: {
    session_id: this.sessionId,
    reason: 'user_request' | 'low_confidence' | 'max_retries',
    partial_ticket: this.draftTicket,
  },
}))
```

业务方监听示例：

```javascript
document.addEventListener('onRepairTicketGenerated', (e) => {
  console.log('工单:', e.detail)  // 直接拿到结构化 JSON
  myApp.submitTicket(e.detail)
})
```

### 3.3 会话状态管理

会话存储在内存字典中（生产可替换为 Redis），TTL 30 分钟：

```python
@dataclass
class Session:
    session_id: str
    client_id: str
    state: AgentState         # 当前对话阶段
    history: list[dict]       # [{role, content}] 完整对话历史
    draft: TicketDraft        # 已收集字段，持续累积更新
    created_at: datetime
    expires_at: datetime
    retry_count: int = 0      # 单字段追问失败次数，达3次触发转人工
```

```python
@dataclass
class TicketDraft:
    description: str | None = None
    normalized_description: str | None = None
    building: str | None = None
    floor: str | None = None
    room: str | None = None
    image_urls: list[str] = field(default_factory=list)
    reporter_name: str | None = None

    def missing_required(self) -> list[str]:
        required = {"description": self.description,
                    "building": self.building,
                    "floor": self.floor}
        return [k for k, v in required.items() if not v]
```

## 四、Agent 对话状态机

### 4.1 状态转移图

```
          ┌─────────┐
          │  INIT   │
          └────┬────┘
               │ init 后自动发欢迎语
          ┌────▼────────┐
          │  GREETING   │ 发出欢迎语，等待用户首次输入
          └────┬────────┘
               │ 收到任意用户消息
          ┌────▼────────┐
          │ COLLECTING  │◄──────────────┐
          └────┬────────┘               │
               │                        │ 字段仍缺失，继续追问
               │ draft.missing_required() == []
               ▼
          ┌────────────┐
          │ CONFIRMING │ 展示工单摘要，询问"以上信息是否正确？"
          └────┬───────┘
               │                    ┌──────────────┐
        ┌──────┴──────┐   用户否认   │  COLLECTING  │
        │用户确认(是)  ├────────────►│ 重新追问变更项│
        └──────┬──────┘             └──────────────┘
               ▼
          ┌──────────┐
          │COMPLETED │ 生成工单 → SSE ticket_ready → 触发 CustomEvent
          └──────────┘

  任何阶段发现 needs_human=true 或 retry_count≥3:
          ┌──────────┐
          │ESCALATED │ 触发 onRequestHumanService CustomEvent
          └──────────┘
```

### 4.2 COLLECTING 阶段核心流程

每轮用户消息进入后执行两步 LLM 调用：

**Step 1 — Extraction Pass（提取新字段，结构化输出）**

```
System: 你是信息提取助手，从用户消息中提取报修相关字段，只输出 JSON，缺失填 null。
        needs_human 仅当用户明确说"转人工""找人"时为 true。

User context:
  当前已知: {draft.to_dict()}
  用户本轮消息: "{user_message}"
  （如有图片已上传，image_url: "{image_url}"）

输出 Schema:
{
  "description": string|null,
  "building": string|null,
  "floor": string|null,
  "room": string|null,
  "needs_human": boolean
}
```

**Step 2 — Reply Pass（生成自然语言回复，流式输出）**

```
System: 你是企业设施报修助手，语气亲切简洁，一次只问一个问题。
        已收集: {updated_draft.to_dict()}
        缺失必填项: {missing_fields}

        规则：
        - 如果 description 缺失，先问问题描述
        - 如果 building 缺失，问"请问是哪栋楼？"
        - 如果 floor 缺失，问"在几楼？"
        - 如果必填项已齐全但没有图片，提示一次"方便拍张照片吗？（可跳过）"
        - 如果已提示拍照，不再重复提示
        - 如果所有必填项已齐全，回复"好的，我来帮您整理一下报修信息…"并结束 COLLECTING

User: [对话历史]
```

### 4.3 CONFIRMING 阶段

必填字段齐全后，执行 RAG 检索，然后让 LLM 生成确认摘要：

```
AI 回复示例：
"好的，我已记录您的报修信息，请确认：
  • 位置：A栋 3楼 302会议室
  • 问题：空调不制冷
  • 图片：已上传1张
  • 故障类型：机电类报修（系统自动判断）
  • 优先级：高

信息无误请回复"确认"，如需修改请告诉我哪里不对。"
```

用户回复"确认"/"是"/"对"/"好的" → 进入 COMPLETED  
用户指出修改点 → 更新 draft，回到 COLLECTING

## 五、RAG 检索服务

### 5.1 数据预处理脚本（`scripts/ingest_tickets.py`）

```
输入：历史工单 CSV（5000条）
  columns: [raw_description, fault_type_code, fault_type_name,
            repair_priority, repair_type]

处理步骤：
  1. 去重、过滤空描述
  2. 批量调用 LLM（batch_size=50）标准化描述：
       Prompt: "将下列报修描述转化为'物理实体+故障现象'格式（不超过15字），
               只输出标准化文本，不解释。"
       原始: "1102单元茶水间的锁坏了，开不了门"
       输出: "门锁损坏，无法开门"
  3. 用 BAAI/bge-large-zh 对标准化描述做向量化
  4. 批量 upsert 到 ChromaDB collection: "historical_tickets"
       vector: embedding
       metadata: { fault_type_code, fault_type_name,
                   repair_priority, repair_type, raw_description }
```

### 5.2 运行时检索流程

```python
async def search_fault(description: str) -> RagResult:
    # 1. 用 LLM 将用户描述标准化（同入库时同等 Prompt）
    normalized = await llm.normalize_problem(description)
    # 2. 向量化
    vec = embedding_model.encode(normalized)
    # 3. ChromaDB 搜索 top-3
    results = chroma_client.query(
        collection_name="historical_tickets",
        query_embeddings=[vec],
        n_results=3,
    )
    # 4. 取得 top-1 作为主要参考
    top = results["metadatas"][0][0]
    score = 1 - results["distances"][0][0]  # 余弦相似度
    return RagResult(
        fault_type={"code": top["fault_type_code"],
                    "displayName": top["fault_type_name"]},
        repair_priority=top["repair_priority"],
        repair_type=top["repair_type"],
        match_score=score,
        confidence="high" if score > 0.85
                  else "medium" if score > 0.65
                  else "low",
    )
```

置信度为 `low` 时，工单仍然生成并发给业务方，同时写入 `fallback_records` 表供人工复核。

## 八、图片存储设计（MinIO）

### 8.1 Bucket 规划

```
Bucket: facility-repairs
  └── 2026/
      └── 05/
          └── 20/
              └── {session_id[:8]}_{uuid4()[:8]}.jpg
```

### 8.2 上传安全策略

- 后端校验 session_id 有效性后才接收文件
- 文件名用 `uuid4` 生成，不暴露原始文件名（防遍历）
- MIME type 白名单：`image/jpeg`、`image/png`、`image/webp`
- 单文件限制 10MB，压缩后实际约 300KB~1MB
- 返回 URL：开发环境直接 HTTP URL；生产环境生成带签名的临时 URL（TTL 72h）

### 8.3 周期清理策略

MinIO 设置 Lifecycle 规则：`facility-repairs` bucket 中超过 **90 天**的对象自动删除。

| 服务 | 镜像 | 端口 | 说明 |
|------|------|------|------|
| `backend` | 自构建 FastAPI | 8000 | 主服务 |

## 十、关键非功能设计

### 10.1 流量与超时

| 场景 | 策略 |
|------|------|
| LLM 调用超时 | 15s 超时，超时后返回 SSE error 事件，前端提示重试 |
| 图片上传超时 | 30s |
| Session 过期 | 30min 不活跃自动清理，前端重新 init |
| 并发限制 | 单 client_id 最多 50 个并发 session |

### 10.2 微信小程序适配说明

Web Component 通过 `<web-view>` 嵌入 H5 页面在小程序中运行时：
- 摄像头：使用小程序原生 `wx.chooseMedia` API，H5 侧通过 `postMessage` 接收图片 base64
- 语音：小程序 `wx.getRecorderManager` 录音 → 前端 Web Speech API 降级为文字输入（需额外适配层）
- CustomEvent 回调：通过 `<web-view>` 的 `message` 事件透传给小程序

### 10.3 错误兜底

| 错误场景 | 处理 |
|----------|------|
| LLM 连续失败 2 次 | 返回固定话术"抱歉系统繁忙，请稍后重试" |
| RAG 检索无结果 | confidence=low，fault_type 填 `{"code":"000","displayName":"其他"}` |
| 用户同一字段追问 3 次仍无效 | 触发转人工，写入 fallback_records |
| 图片上传失败 | 前端提示"上传失败，可跳过图片继续报修" |

## 十一、开发分阶段计划

| 阶段 | 内容 | 产出 |
|------|------|------|
| **P0** | 后端骨架 + Agent状态机 + LLM调用 | 可用 curl 完成完整报修对话，产出工单 JSON |
| **P1** | RAG 数据入库 + 检索接入 | fault_type / priority 自动判断 |
| **P2** | MinIO 图片存储 + Upload 接口 | 支持图片上传 |
| **P3** | Web Component 基础版（文字） | 浏览器内完整流程 |
| **P4** | 语音输入 + TTS 输出 | 语音报修体验 |
| **P5** | docker-compose 一键部署 + 小程序适配 | 交付生产 |
